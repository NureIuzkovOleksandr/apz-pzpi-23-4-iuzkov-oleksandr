1 BUSINESS REQUIREMENTS (БІЗНЕС-ВИМОГИ)

1.1 Аналіз проблеми

     У сучасних приміщеннях немає централізованого контролю за температурою та вологістю, що призводить до дискомфорту, втрат енергії та складнощів у відстеженні змін клімату. Існуючі пристрої часто не взаємодіють між собою та не надають аналітики в реальному часі.
Основні проблеми:
1) Відсутність автоматичного відстеження показників.
2) Неможливість централізованого управління пристроями.
3) Обмежені можливості аналітики та оповіщень.
4) Неоптимальне використання енергії через ручне регулювання.
1.2 Аналіз аналогів

    Серед наявних рішень виділяються Nest, Netatmo, Xiaomi Smart Home. Вони дозволяють контролювати клімат, але часто не інтегруються з іншими IoT- пристроями та не забезпечують гнучких API для аналітики.
Розроблювана система надасть:
1) централізоване керування даними з усіх сенсорів,
2) автоматичне управління кліматичними пристроями,
3) веб- та мобільні інтерфейси для моніторингу та управління,
4) зберігання історії змін та аналітику в реальному часі,
5) можливість масштабування для декількох кімнат чи приміщень.

3 SCOPE AND LIMITATIONS (РАМКИ ТА ОБМЕЖЕННЯ ПРОЄКТУ)

3.1 Scope of Initial Release (Рамки первинного випуску)

     Первинний реліз системи SmartClimate охоплює повний цикл роботи платформи: від збору даних на рівні IoT-обладнання до відображення інформації у веб- та мобільних інтерфейсах. До складу релізу входять серверна частина, IoT- клієнт, веб-клієнт та мобільний застосунок.

3.1.1 Серверна частина

     Сервер реалізовано на Python 3 з використанням фреймворку FastAPI та бази даних PostgreSQL. Він є центральним елементом системи та виконує такі функції:
     1. Реєстрація та автентифікація користувачів (JWT-токени); ведення журналу дій.
     2. Приймання телеметрії від IoT-клієнтів через протокол MQTT, валідація та збереження показників у базі даних.
     3. Автоматичне керування кліматичними пристроями на основі заданих порогових значень температури та вологості.
     4. Генерація аналітичних звітів і графіків змін клімату за довільний часовий проміжок.
     5. Відправлення push- та email-сповіщень при виявленні аномальних показників.
     6. REST API з автоматичною документацією Swagger UI для взаємодії з клієнтами.


3.1.2 IoT-клієнт

     IoT-клієнт реалізовано на мікроконтролері ESP32 з програмним забезпеченням на MicroPython. ESP32 підключається до локальної Wi-Fi-мережі та взаємодіє з сервером через брокер MQTT. Виконує такі функції:
     1. Циклічний збір показників температури та вологості з підключених сенсорів (DHT22 / SHT31) через цифровий інтерфейс.
     2. Передача телеметрії на сервер через протокол MQTT (бібліотека umqtt.simple).
     
     3. Виконання команд сервера: вмикання/вимикання кондиціонера, обігрівача або зволожувача через GPIO-піни та реле-модуль.
     4. Самодіагностика: виявлення втрати з'єднання (Wi-Fi / MQTT) з автоматичним перепідключенням та передачею сигналів про помилки на сервер.



3.1.3 Веб-клієнт

     Веб-застосунок розроблено на базі React (HTML5, CSS3, JavaScript) та взаємодіє з сервером через REST API. Забезпечує такі можливості:
     1. Відображення поточних показників температури та вологості в режимі реального часу.
     2. Інтерактивні графіки динаміки змін клімату (погодинно, за добу, тиждень, місяць).
     3. Налаштування порогових значень для автоматичного спрацьовування пристроїв.
     4. Ручне керування кліматичними пристроями безпосередньо з браузера.
     5. Перегляд журналу подій та сповіщень, управління профілем користувача.


3.1.4 Мобільний клієнт

     Мобільний додаток розроблено на Kotlin для платформи Android з використанням Jetpack Compose для побудови UI та Retrofit для взаємодії з REST API сервера. Включає такий функціонал:

     1. Перегляд актуальних даних з будь-якого місця через мобільний інтернет.
     2. Push-сповіщення про критичні зміни клімату (перевищення порогів).
3. Віддалене ввімкнення та вимкнення кліматичних пристроїв.

     4. Синхронізація стану пристроїв та показників у реальному часі з сервером.


     Робота системи в межах первинного релізу реалізується за наступним сценарієм: IoT-клієнт на ESP32 збирає дані та передає їх через MQTT на сервер; сервер обробляє ці дані, зберігає в PostgreSQL і, за потреби, генерує сповіщення та команди керування; веб-клієнт і мобільний застосунок надають користувачу інтерфейс для контролю та управління.
3.2 Scope of Subsequent Releases (Рамки наступних випусків)

     У наступних релізах системи передбачається поетапне розширення функціональних можливостей із фокусом на підвищення рівня персоналізації, аналітики та автоматизації. Ключовим напрямом розвитку є інтеграція технологій штучного інтелекту.
3.2.1 Застосування — як ШІ покращить роботу системи

     Поточна система працює на основі статичних порогових значень, що не враховують індивідуальні вподобання користувача, зовнішні погодні умови чи час доби. Інтеграція ШІ дозволить реалізувати такі сценарії:
     1. Предиктивне регулювання клімату. Модель машинного навчання аналізуватиме часові ряди температури та вологості й
прогнозуватиме майбутні відхилення. Замість реактивного спрацьовування

система проактивно вмикатиме пристрій заздалегідь, що зменшить коливання та споживання енергії.
     2. Персоналізований комфорт. ШІ вивчатиме звички користувача (час пробудження, графік роботи, бажані температурні діапазони в різний час доби) та автоматично адаптуватиме пороги без ручного налаштування.
3. Автоматична діагностика аномалій. Алгоритм виявлення
аномалій відрізнятиме нормальні коливання клімату від нетипових подій (несправність сенсора, відкрите вікно, поломка пристрою) та сповіщатиме користувача без хибних тривог.



3.2.2 Збирання даних для навчання моделей

     Система вже з першого релізу накопичує дані, які стануть основою для навчання ШІ-моделей. Типи даних та способи їх збору:
     1. Телеметрія сенсорів. Показники температури та вологості зберігаються в PostgreSQL з часовою міткою та ідентифікатором
приміщення. Це формує часові ряди з кроком до 30 секунд — достатня гранулярність для навчання регресійних та прогностичних моделей.
     2. Дії користувача. Журнал подій фіксує, коли і при яких показниках клімату користувач вручну змінював стан пристроїв або
коригував порогові значення. Це є прямим сигналом переваг (label) для навчання моделі персоналізації.
3. Зовнішні дані. Через відкритий API погоди (наприклад,
OpenWeatherMap) сервер збиратиме зовнішню температуру, вологість та прогноз. Ці дані збагачуватимуть вхідні ознаки моделі для врахування зовнішнього середовища.

     4. Технічна реалізація збору. Усі дані зберігаються у структурованих записах бази даних. Для підготовки датасетів передбачений окремий модуль експорту у форматах CSV/Parquet для передачі в ML- pipeline.



3.2.3 Навчання та використання моделей

     Вибір	підходу	до	ШІ	залежить	від	конкретної	задачі.	Планується комбінований підхід:
     1. Власне навчання (prediction & anomaly detection). Для прогнозування часових рядів та виявлення аномалій планується навчання власних моделей (LSTM, Prophet, Isolation Forest) на накопичених даних конкретної системи. Кліматичні патерни кожного приміщення унікальні й
визначаються архітектурою будівлі та типом опалення — загальні моделі не враховують цю специфіку.
     2. Донавчання (fine-tuning) для персоналізації. Базова модель рекомендацій ініціалізується на агрегованих даних усіх користувачів, а
потім донавчається під конкретного користувача методами transfer learning. Це скорочує час виходу на якісні рекомендації для нових користувачів.
3. Готовий API для природньомовних запитів. Для текстового
асистента («Встанови комфортну температуру», «Покажи звіт за тиждень») планується інтеграція зовнішнього LLM API. Власне навчання LLM
недоцільне через надмірні ресурсні витрати при відносно вузьких сценаріях використання.
3.3 Limitations and Exclusions (Обмеження та винятки)

     Незважаючи на широкий функціонал, система SmartClimate має свідомо визначені обмеження та виключення.

3.3.1 Технічні та архітектурні обмеження

     1. Обмежена кількість приміщень в одному обліковому записі. Архітектура першого релізу розрахована на моніторинг до 10 кімнат (10 IoT-клієнтів на базі ESP32) в рамках одного акаунта. Масштабування на рівні підприємства виходить за межі початкового скоупу.
2. Відсутність офлайн-режиму. Усі рішення про керування
пристроями приймаються централізовано на сервері. За відсутності мережі IoT-клієнт не здатен самостійно підтримувати кліматичні пороги.
     3. Підтримка обмеженого переліку протоколів. Система підтримує лише MQTT. Пристрої з протоколами Zigbee, Z-Wave або Modbus не
підтримуються в першому релізі.

     4. Залежність від хмарної інфраструктури. Система не передбачає повністю локального (on-premise) розгортання без хмари, що може бути критичним для замовників з обмеженнями щодо передачі даних.



3.3.2 Функціональні виключення

     1. Управління системою вентиляції та рекуперації. Незважаючи на технічну можливість підключення таких пристроїв, складна логіка
управління вентиляцією навмисно виключена з початкового релізу задля спрощення MVP.
     2. ШІ-функції у першому релізі. Прогнозування та персоналізація заплановані лише на наступні релізи, що дозволяє швидше вивести продукт на ринок і почати збір даних для навчання моделей.
     3. Мультикористувацьке управління одним приміщенням. Перший реліз не підтримує спільний доступ кількох рівноправних користувачів до
     
одного приміщення. Реалізація системи ролей і прав відкладена до наступного релізу.
4. Інтеграція з платформами розумного будинку (Google Home,
Amazon Alexa). Офіційних конекторів у першому релізі немає — їх розробка потребує проходження сертифікаційних програм.

4.3 Опис програмних засобів та інструментів

    Для реалізації системи моніторингу клімату в приміщенні використовується сучасний технологічний стек, що забезпечує надійність, масштабованість та інтеграцію з IoT-пристроями.
Серверна частина
1) Мова програмування: Python 3
2) Фреймворк: FastAPI — дозволяє створювати швидкий і безпечний REST API для обміну даними між веб- та мобільними клієнтами та IoT-модулями. API реалізує основні процеси системи: авторизацію користувачів, збір та обробку даних з сенсорів, управління кліматичними пристроями, аналітику та формування звітів.
3) Система керування базами даних: PostgreSQL, забезпечує надійне зберігання історії показників, підтримку транзакцій та можливість виконання складних аналітичних запитів.
4) Середовище розробки: VS Code — основне IDE для розробки серверної логіки, тестування API та налагодження.
5) Контроль версій: Git. Репозиторій розміщено на GitHub для спільної роботи, відстеження змін та історії розробки.
Веб- та мобільні клієнти
1) Веб-клієнт:
a) Технології: React, HTML5, CSS3, JavaScript
b) Призначення: перегляд даних з сенсорів, керування кліматичними пристроями, формування графіків та звітів.
c) Взаємодія із сервером здійснюється через REST API.
2) Мобільний клієнт:
a) Розробка на базі React Native для кросплатформенності (Android/iOS).
b) Призначення: віддалений доступ до даних, керування пристроями та отримання повідомлень про критичні показники.
Інтеграція з IoT-модулем
1) Апаратна платформа: ESP32. Використовується як контролер для підключення сенсорів температури та вологості, а також керування кліматичними пристроями.
2) Програмне забезпечення на IoT:
a) Мова: Python 3
b) Бібліотеки: paho-mqtt, RPi.GPIO, requests
c) Призначення: збір даних сенсорів, виконання команд від сервера та передача результатів на центральний сервер.
3) Протокол взаємодії: MQTT Забезпечує надійну публікацію та підписку для обміну даними між сервером та IoT-пристроями в режимі реального часу.
Тестування та документація
1) Postman — для перевірки REST API та тестування сценаріїв взаємодії сервер-клієнт-IoT.
2) Swagger UI (OpenAPI) — для автоматичної генерації документації для REST API та наочна візуалізація доступних методів.

Середовище розгортання
1) ОС сервера: Windows Server.
2) Хмарна інфраструктура: Microsoft Azure — для хостингу бази даних, серверного API та забезпечення масштабованості.
