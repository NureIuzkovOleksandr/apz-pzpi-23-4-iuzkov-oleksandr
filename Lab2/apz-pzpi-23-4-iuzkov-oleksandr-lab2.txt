2 ЗАВДАННЯ

     На цій лабораторній роботі потрібно розробити мобільний програмний застосунок для будь-якої поширеної на сьогодні мобільної платформи. Мобільний застосунок повинен бути частиною системи і повинен взаємодіяти з нею. Для реалізації програмного застосунку можна використовувати технології для реалізації програмного забезпечення для мобільних платформ, вказані лектором, а саме Kotlin для Android та Swift для iOS. Звіт повинен містити опис прийнятих інженерних рішень. У ньому повинна бути показана будова мобільного застосунку та його взаємодія з іншими частинами системи.


3 ОПИС ВИКОНАНОЇ РОБОТИ

     3.1 Опис роботи

     Умови завдання передбачали використання нативних технологій — Kotlin для Android або Swift для iOS. Проте специфіка проєкту SmartClimate поставила практичну вимогу: мобільний клієнт має однаково добре працювати на обох платформах, оскільки кінцеві користувачі системи моніторингу мікроклімату не обмежені одним виробником пристроїв. Написання та підтримка двох окремих кодових баз для MVP є невиправданою витратою ресурсів.

     Після порівняння кросплатформних альтернатив, для реалізації було обрано Flutter з мовою Dart. Головна технічна перевага, що вирішила вибір: Flutter компілюється у нативний ARM-код і відмальовує власний UI через движок Skia, не покладаючись на нативні компоненти платформи. Це гарантує однакову поведінку інтерфейсу на Android та iOS без платформозалежних обхідних рішень.

     Додатковими факторами стали: наявність вбудованого пакету http для роботи з REST API сервера SmartClimate, пакет flutter_secure_storage для безпечного зберігання JWT-токена в Android Keystore / iOS Keychain, а також механізм Hot Reload, що суттєво скорочує час ітерації під час розробки UI.
     
     3.2 Структура проєкту та шари застосунку

     Вся кодова база зосереджена у каталозі lib/ та поділена на чотири функціональних пакети:

     1) screens/ — містить 15 екранних класів, кожен відповідає за єдиний екран і нічого не знає про логіку суміжних екранів.
     2) services/ — єдиний клас ApiService, що інкапсулює всі HTTP-запити до сервера. Решта застосунку не знає деталей мережевої взаємодії.
     3) state/ — клас AppState на базі ChangeNotifier. Зберігає глобальний стан, доступний будь-якому екрану через Provider без передачі параметрів у конструкторах.
     4) l10n/ — клас AppLocalizations з делегатом. Відповідає за переклад рядків інтерфейсу залежно від поточної локалі.

     Точка входу main.dart ініціалізує ApiService асинхронно (зчитує збережену URL сервера з SharedPreferences), після чого реєструє AppState у кореневому ChangeNotifierProvider і запускає MaterialApp з визначеною таблицею маршрутів.

     3.3 Сервісний шар: ApiService

     ApiService є єдиним місцем у застосунку, де відбувається мережева взаємодія. Клас не є синглтоном у класичному розумінні — він створюється через фабричний метод ApiService.create(), який асинхронно відновлює раніше збережену URL сервера. Це дозволяє підключатися до різних середовищ: локального емулятора (10.0.2.2:8000), фізичного пристрою в LAN або хмарного деплою — без перекомпіляції застосунку.

     JWT access_token після успішного входу записується у FlutterSecureStorage. На Android це означає зберігання у зашифрованому файлі, захищеному Android Keystore; на iOS — у Keychain. Такий підхід унеможливлює зчитування токена іншими застосунками навіть на пристроях без root/jailbreak.

     Для запитів, що змінюють стан сервера (POST, PUT, PATCH), реалізовано уніфікований метод requestWithResponse(), який повертає як HTTP-статус, так і декодоване тіло відповіді. Це дозволяє екранам розрізняти помилки валідації (422) від серверних помилок (500) і показувати користувачу конкретне повідомлення замість загального «щось пішло не так».

     3.4 Управління станом

     У якості підходу до управління станом обрано Provider — офіційно рекомендований пакет команди Flutter для більшості застосунків рівня MVP. Складніші альтернативи (BLoC, Riverpod) надають більше можливостей, але вносять додаткову архітектурну складність, невиправдану для поточного масштабу проєкту.

     AppState реалізує лише ту частину глобального стану, яка справді є глобальною: поточну локаль інтерфейсу. Виклик setLocale() оновлює поле _locale та викликає notifyListeners(), що ініціює перебудову MaterialApp і миттєво застосовує новий переклад без перезапуску. Стан окремих екранів (списки кімнат, показники сенсорів тощо) зберігається локально у StatefulWidget кожного екрана і не забруднює глобальний простір стану.

     3.5 Опис реалізованих екранів

     Застосунок налічує 15 екранів, логічно згрупованих у шість функціональних блоків.

     Автентифікація. 
     LoginScreen будує форму входу та виконує двоетапну спробу логіну: спочатку JSON-запит до /api/auth/login, у разі невдачі — form-encoded запит до /api/auth/login/form. Такий підхід забезпечує сумісність із різними конфігураціями серверного OAuth2. RegisterScreen збирає дані нового користувача та надсилає їх на /api/users/register.

     Кімнати. 
     RoomsScreen завантажує список кімнат через GET /api/rooms і виводить їх у вигляді карток. Свайп картки вліво видаляє кімнату через DELETE /api/rooms/{id}. RoomFormScreen використовується як для створення нової кімнати (POST), так і для редагування існуючої (PUT) — залежно від того, чи передано об'єкт кімнати через аргументи маршруту.

     Сенсори. 
     SensorsScreen виводить перелік датчиків із зазначенням кімнати прив'язки. SensorDetailScreen отримує історію показань через GET /api/sensors/{id}/readings і відображає їх у вигляді прокручуваного списку з часовими мітками. SensorFormScreen реалізує CRUD-форму для датчиків.

     Кліматичні пристрої. 
     ClimateDevicesScreen показує список пристроїв з поточним статусом. Кнопки «Увімкнути» та «Вимкнути» відправляють команди через POST /api/climate-devices/{id}/control з параметром command у тілі запиту. DeviceFormScreen забезпечує додавання та редагування пристроїв.

     Порогові значення. 
     ThresholdsScreen відображає налаштовані пороги температури та вологості для кожної кімнати, перевищення яких автоматично активує кліматичні пристрої на сервері. ThresholdFormScreen дозволяє задати мінімальне та максимальне значення для кожного параметра.

     Сповіщення, профіль, налаштування. 
     AlertsScreen завантажує сповіщення (GET /api/alerts) із можливістю позначення як прочитаних та видалення. ProfileScreen відображає дані облікового запису і дозволяє змінити їх або пароль. SettingsScreen дає змогу оновити URL сервера (з перевіркою з'єднання через /health) та переключити мову інтерфейсу між англійською та українською.

     3.6 Взаємодія мобільного клієнта з системою SmartClimate

     Місце мобільного застосунку в загальній архітектурі системи визначається так: він є одним із двох клієнтів серверної частини (поряд із веб-клієнтом) і не взаємодіє з IoT-рівнем безпосередньо. Вся телеметрія надходить від ESP32-пристроїв на сервер через MQTT; застосунок отримує вже оброблені дані через REST API.

     Типовий сценарій взаємодії виглядає так: ESP32 фіксує перевищення порогу вологості і надсилає показник на сервер; сервер порівнює значення з налаштованим порогом, вмикає зволожувач командою через GPIO та записує сповіщення в базу даних; мобільний застосунок при наступному відкритті AlertsScreen завантажує це сповіщення і показує його користувачу. Таким чином мобільний застосунок забезпечує людино-машинний інтерфейс для всього циклу моніторингу та управління.
     
     
     3.7 Використання сервісів штучного інтелекту:

     3.7.1 Запит 1

     Назва інструменту та версія: Claude Sonnet 4.5.
     Мета використання: З'ясувати практичні відмінності між Flutter, React Native та Kotlin Multiplatform з точки зору інтеграції з REST API та зберігання секретів на пристрої.
     Внесок автора: Отримані матеріали зіставлено з вимогами проєкту SmartClimate. Аргументовано вибір Flutter на основі нативної компіляції, наявності flutter_secure_storage та відсутності залежності від JavaScript Bridge.

     3.7.2 Запит 2

     Назва інструменту та версія: Claude Sonnet 4.5.
     Мета використання: Отримати рекомендації щодо організації HTTP-клієнта у Flutter: обробка помилок, зберігання токенів, конфігурація базової URL для різних середовищ розгортання.
     Внесок автора: Запропонований підхід адаптовано під ендпоїнти FastAPI-сервера SmartClimate. Реалізовано фабричний метод create() з відновленням URL із SharedPreferences, дворівневу логіку логіну та уніфікований requestWithResponse().


4 ВИСНОВКИ

     У результаті виконання лабораторної роботи для системи SmartClimate розроблено повнофункціональний мобільний застосунок на платформі Flutter. Прийняте рішення відступити від нативних технологій на користь кросплатформного підходу обґрунтоване практичною необхідністю охопити обидві мобільні платформи при обмежених ресурсах MVP.

     Реалізований застосунок інтегрується з серверною частиною SmartClimate через REST API з JWT-авторизацією і надає користувачу повний набір інструментів для роботи з системою: перегляд телеметрії сенсорів, керування кліматичними пристроями, налаштування порогів автоматичного спрацьовування, отримання сповіщень про аномалії та управління обліковим записом.

     З архітектурної точки зору застосунок демонструє чітке розмежування відповідальностей між шарами Presentation, Service та State, що спрощує подальше розширення функціоналу — зокрема, додавання push-сповіщень через FCM та графічної візуалізації телеметрії, заплановані в наступних релізах системи.
