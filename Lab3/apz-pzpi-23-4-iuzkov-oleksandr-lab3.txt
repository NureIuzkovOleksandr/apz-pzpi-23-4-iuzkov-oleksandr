2 ЗАВДАННЯ
     У межах лабораторної роботи необхідно розробити клієнтську частину програмної системи Climate Monitor — застосунку для контролю мікроклімату в приміщеннях. Клієнтська частина повинна забезпечувати роботу у веб-режимі та підтримувати повноцінне адміністрування системи через окремий інтерфейс.
     Інтерфейс звичайного користувача надає можливість переглядати дані з датчиків, керувати пристроями та отримувати сповіщення про перевищення порогових значень мікроклімату. Адміністративний інтерфейс забезпечує управління обліковими записами користувачів, імпорт та експорт налаштувань, резервне копіювання даних, а також перегляд системних журналів.
     Обидва інтерфейси повинні підтримувати інтернаціоналізацію та локалізацію: перемикання між українською та англійською мовами без перезавантаження сторінки, відображення дат і часу у форматі відповідного регіону, коректне сортування кириличних і латинських рядків з урахуванням поточної локалі.


3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Опис прийнятих інженерних рішень
     Клієнтська частина системи Climate Monitor реалізована як односторінковий застосунок (SPA) на основі React 18. Для маршрутизації використовується React Router v6, що забезпечує декларативне визначення маршрутів та захист захищених сторінок через компонент-обгортку PrivateRoute. HTTP-взаємодія з REST API бекенду здійснюється через бібліотеку Axios з централізованим перехопленням відповідей.
     Інтернаціоналізація реалізована за допомогою бібліотеки i18next з плагіном react-i18next. Файли перекладів зберігаються у форматі JSON та охоплюють усі елементи інтерфейсу. Поточна мова зберігається у localStorage під ключем «climate_lang» і відновлюється при повторному відкритті застосунку. Формат відображення дат і часу формується через стандартний API Intl.DateTimeFormat відповідно до обраної локалі: дд.мм.рррр для української та мм/дд/рррр для англійської.
     Автентифікація побудована на основі JWT-токенів. Після успішного входу токен зберігається у localStorage та автоматично додається до заголовків усіх запитів через Axios-інтерцептор. Відповідь з кодом 401 ініціює автоматичний вихід із системи та перенаправлення на сторінку входу, а відповідь з кодом 403 відображає сторінку відмови у доступі.
     Рольова модель передбачає два типи облікових записів: звичайний користувач та адміністратор. Доступ до захищених маршрутів перевіряється на рівні маршрутизатора: компонент PrivateRoute зчитує роль із декодованого JWT-токена і перенаправляє на відповідний розділ або на сторінку відмови у доступі.
     Стани завантаження відображаються через skeleton-елементи для таблиць і карток та через spinner для форм і кнопок підтвердження. Операції видалення вимагають підтвердження через модальне діалогове вікно. Серверні помилки відображаються через toast-сповіщення тривалістю три секунди. Усі форми містять інлайн-валідацію з підказками безпосередньо під полями вводу.
     3.2 Опис взаємодії компонентів
     Застосунок складається з трьох функціональних блоків: публічного розділу (сторінка входу), розділу звичайного користувача та адміністративного розділу. Точкою входу є компонент App, який визначає кореневий маршрутизатор і підключає провайдери контексту автентифікації та локалізації.
     Компонент Sidebar формує навігаційне меню залежно від ролі поточного користувача. Для звичайного користувача відображаються посилання на дашборд, пристрої, датчики, приміщення, порогові значення, сповіщення, профіль та налаштування. Для адміністратора — панель адміністратора, управління користувачами, системні журнали та системні налаштування.
     Взаємодія з бекендом централізована у модулі api.ts, який експортує сконфігурований екземпляр Axios з базовою URL-адресою REST API. Усі сторінки імпортують функції запитів із цього модуля, що забезпечує єдину точку конфігурації та обробки помилок. Типи даних, що передаються між компонентами та API, описані у файлі types.ts засобами TypeScript.
     Сторінки користувацького розділу реалізують такі функції: DashboardPage відображає зведену панель з актуальними показниками датчиків; DevicesPage, DeviceCreatePage та DeviceEditPage забезпечують перегляд, додавання та редагування IoT-пристроїв; SensorsPage, SensorCreatePage та SensorEditPage — аналогічний цикл для датчиків; RoomsPage та RoomFormPage — управління приміщеннями; ThresholdsPage — налаштування порогових значень параметрів мікроклімату; AlertsPage — перегляд активних сповіщень; ProfilePage та SettingsPage — особисті налаштування облікового запису та преференції відображення.
     Адміністративний розділ включає чотири сторінки. AdminDashboardPage відображає загальну статистику системи. AdminUsersPage забезпечує повний цикл управління обліковими записами: створення нових користувачів, редагування їх даних та ролей, а також видалення облікових записів. AdminSystemPage надає можливість імпортувати та експортувати налаштування системи у вигляді файлів, а також створювати резервні копії поточної конфігурації. AdminLogsPage дозволяє переглядати та фільтрувати системні журнали подій.
     3.3 Використання сервісів штучного інтелекту
     Запит 1
     Назва інструменту та версія: Claude Sonnet 4.6.
     Мета використання: вивчення підходів до організації захищеної маршрутизації у React-застосунках на основі JWT та рольової моделі доступу.
     Внесок автора: отримані матеріали проаналізовано та адаптовано для реалізації компонента PrivateRoute з перевіркою ролі користувача і перенаправленням на відповідні сторінки.
     Запит 2
     Назва інструменту та версія: Claude Sonnet 4.6.
     Мета використання: з'ясування принципів налаштування i18next у проєктах TypeScript з динамічним перемиканням мови без перезавантаження сторінки.
     Внесок автора: на основі наданих прикладів розроблено конфігурацію i18n.ts та підключено провайдер перекладів до кореневого компонента застосунку.
     Запит 3
     Назва інструменту та версія: Claude Sonnet 4.6.
     Мета використання: отримання рекомендацій щодо структурування централізованого API-модуля на Axios для TypeScript-проєктів.
     Внесок автора: запропоновану структуру переопрацьовано під специфіку системи Climate Monitor — додано типізацію через types.ts та реалізовано інтерцептори для обробки помилок автентифікації.


4 ВИСНОВКИ
     У ході виконання лабораторної роботи розроблено клієнтську частину системи Climate Monitor — застосунку для контролю мікроклімату в приміщеннях. Реалізовано окремі інтерфейси для звичайних користувачів та адміністраторів із захищеною маршрутизацією на основі JWT-токенів і рольової моделі доступу.
     Технологічну основу застосунку складають React 18, React Router v6, Axios та TypeScript. Підтримку двох мов інтерфейсу — української та англійської — забезпечено через i18next з динамічним перемиканням без перезавантаження сторінки. Формати дат, часу та порядок сортування рядків адаптовано до обраної локалі.
     Адміністративний розділ надає повний цикл управління обліковими записами користувачів: створення, редагування та видалення. Реалізовано функції імпорту та експорту налаштувань системи, створення резервних копій конфігурації, а також перегляд і фільтрацію системних журналів. На всіх сторінках застосовано єдиний підхід до обробки помилок, валідації форм та відображення стану завантаження.
